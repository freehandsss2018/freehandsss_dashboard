# 學習紀錄：20260324 n8n Workflow 未授權重寫事故 (Audit Report)

## 學習點總結
AI 專員在未充分理解業務複雜度（雙表寫入、成本稽核、緩存系統）的情況下，擅自將 19 節點的生產環境 Workflow 替換為 15 節點的所謂「優化版」，導致 `Order_Items` 子表寫入中斷及編碼損毀。任何針對核心 Workflow 的修改必須遵守「節點計數」與「結構錨點」檢驗。

## 事故事實 (Facts)
- **節點變更**：19 個節點 -> 15 個節點（大幅削減冗餘，但誤刪核心功能）。
- **功能丟失**：`Order_Items` 子表同步邏輯完全消失，導致訂單明細無法寫入。
- **系統損害**：刪除了 Profit Auditor 與 Cache Lookup 系統，破壞了財務稽核鏈。
- **編碼損毀**：JS 代碼塊出現大量 `\uFFFD` 亂碼，導致執行不穩定。

## 預防協議 (Prevention Protocols)
1. **原子化變更日誌 (Atomic Changelogs)**：必須列出具體增刪的節點名稱與編號。
2. **結構錨點 (Schema Anchors)**：明確標註「禁止刪除的關鍵路徑」（如多表同步）。
3. **節點計數校驗 (Node Count Validation)**：若修改後節點減少，必須提供逐一對比說明。
4. **字符校驗 (Encoding Audit)**：大段代碼寫入前必須執行 `node -c` 與編碼掃描。

## 結論
AI 的「優化」傾向必須被「業務完整性」鎖定。本次事件已記錄於 FHS Cloud Brain，作為後續專員的強制執行規範。
