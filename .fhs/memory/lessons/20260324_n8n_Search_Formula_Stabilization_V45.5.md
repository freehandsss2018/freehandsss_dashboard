# 20260324_n8n_V45.7_Four_Layer_Onion_Crisis

### 學習點總結：四層洋蔥錯誤 (Four-Layer Onion)
本次任務揭示了極具隱蔽性的「聯合金叉」故障：一個引擎層級的 Bug 掩蓋了多個代碼/配置層級的錯誤。當修好第一層時，第二層才顯現，造成「修補無效」的錯覺。

**核心教訓**：
1.  **節點版本黑名單**：禁用 **n8n IF 節點 v2.3**。該版本存在邏輯路由 Bug（永遠走 Branch 0）。所有新開發應統一使用 **Switch 節點 v1** 作為標準。
2.  **JS 代碼完整性稽核**：在大量 JSON 合併或環境遷移後，必須逐一掃描 `jsCode` 欄位。本次事故中 7 個 Code 節點代碼莫名遺失，是導致流程「空轉」的主因。

### 技術細節 (四層嵌套故障)
*   **第一層：IF Node Bug** (v2.3) -> 導致無論條件為何，始終強制執行一條路徑，掩蓋了邏輯判斷。
*   **第二層：Code Node Data Loss** -> 從 V4 備份還原時發現多個關鍵節點（如利潤計算）的 JS 程式碼為空。
*   **第三層：Cache ENOENT** -> `products.json` 遺失，導致讀取節點報錯中止流程。
*   **第四層：Airtable 404** -> 由於前級數據解析失敗，導致傳入 `delete` 路徑的 ID 為空或格式錯誤。

### 技術演進 (V45.7.1)
- **取代方案**：以 `Switch Node (v1)` 全面替換 `IF Node (v2.3)`。
- **數據洗淨**：新增 `Input Normalizer` (V47) 節點，實作 3 種常見 Payload 格式的自動展平。
- **容錯機制**：在 Cache 讀取節點開啟 `Continue On Fail`。

### 防止再發建議
1.  **「疑病從換」**：若同一個節點在配置正確的情況下連續 2 次測試異常，立即刪除並重新拖入新節點，不要嘗試修補版本 Bug。
2.  **備份回溯**：在處理大型流程崩潰時，首要動作是與最近一次 **「完整運作過的 JSON 備份」** 進行 `jsCode` 的 Diff 比對。
