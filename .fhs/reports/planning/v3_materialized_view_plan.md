## v3.0 — Materialized View (Future)

### Goal
Eliminate the dual-path inconsistency by making `order_items` the **single source of truth** for display text. A materialized view (`mv_order_display_text`) regenerates the full IG-format template from `order_items` on demand.

### Key Design

```sql
-- Trigger: rebuild text snapshot whenever order_items changes
CREATE OR REPLACE FUNCTION refresh_order_display_text()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  UPDATE orders
  SET full_order_text = (
    SELECT build_ig_template(NEW.order_fhs_id)
  ),
  full_order_text_a = (...立體擺設 section...),
  full_order_text_b = (...金屬 sections...),
  is_text_overridden = false,
  updated_at = NOW()
  WHERE order_id = NEW.order_fhs_id;
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_refresh_order_text
AFTER INSERT OR UPDATE OR DELETE ON order_items
FOR EACH ROW EXECUTE FUNCTION refresh_order_display_text();
```

### Benefits vs v2
- Mode 1 manual text edit becomes unnecessary (overview always reflects `order_items`)
- `is_text_overridden` flag retired (no longer needed)
- `fhsRegenBtn` retired
- `saveOrderText()` retired or repurposed as admin override only
- n8n `sync_order_to_mirror` migration 0018 guard retired

### Prerequisites
1. `build_ig_template()` PostgreSQL function must reproduce the exact Instagram template format currently generated in JavaScript
2. All `order_items` must have `engraving_text`, `specification`, `quantity` fully populated (no NULL gaps)
3. Fat Mo acceptance test: compare 100-record snapshot before/after migration

### Migration Path
1. Create `build_ig_template(order_id TEXT) RETURNS TEXT` function
2. Test on staging: compare output vs current `full_order_text` for all 26+ orders
3. Create trigger `trg_refresh_order_text`
4. Run `UPDATE orders SET full_order_text = build_ig_template(order_id)` for all rows
5. Remove `is_text_overridden` guard from `sync_order_to_mirror` (migration 0019)
6. Remove Mode 1 `saveOrderText()` override path from Dashboard (or demote to admin-only)
7. Remove `fhsRegenBtn` and `fhsOverrideBadge` from Modal

### Risk
- R1: PostgreSQL IG template must exactly match the JavaScript template — any whitespace/emoji difference will break Fat Mo's copy-paste workflow
- R2: Trigger overhead on `order_items` write (acceptable at FHS scale, ~30 orders/month)

### Trigger Condition
Start v3 implementation when:
- Mode 2 is in stable production use (no bugs for 2+ weeks)
- Fat Mo reports friction with Mode 1/Mode 2 coexistence
- OR: new product categories require IG template changes that would need manual re-sync

## Notes from v2 Implementation

- The structural text generated by `save_structured_order_items` RPC is intentionally a simplified summary (`[立體擺設] spec xQty (刻字: ...)`) rather than the full IG template. This is because replicating the full template in SQL is complex; the RPC sets `is_text_overridden = false` to signal readiness for n8n proper regeneration.
- `full_order_text_a` and `full_order_text_b` are written by the Dashboard `sbSyncOrder` path, NOT by `sync_order_to_mirror` (n8n). This split-write design must be preserved in v3.
