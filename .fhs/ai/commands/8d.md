# /8d — 八維度架構分析＋自我迭代（In-Chat 輕量版）

**用途**：對一個「上述」提案（前一則訊息的草案/方向）在**同一則回覆內**跑完整迭代：草案 → 實施計畫草案 → 自我批評（3個弱點） → 更好的版本 v2 → 八維度檢查 → 三平台顧慮。純文字輸出，**不呼叫任何外部腳本/subagent，不產生 artifacts**，比 `/cl-flow-fast` 更輕。
**觸發指令**：`/8d [上述內容 或 具體任務]`（省略參數 = 針對緊接在此指令前、對話中最後一則 AI 提案或方案做迭代）
**版本**：v1.0.0（2026-07-07，S153 新建）
**來源**：Fat Mo 於 90 個歷史 session 中手打此段分析框架 36 次（`/fhs-usage-audit` 首跑最高頻發現），落成指令消滅重複輸入。

## 與既有管道的分界（避免重複造輪）

| | `/rp` | `/cl-flow-fast` | `/8d` |
|---|---|---|---|
| 性質 | 精煉問題成 XML，**不產出解答草案** | 呼叫外部 Runner（Gemini AG）產出實作計劃，需 Gate 審閱 | 純文字迭代，**Claude 自己**在單則回覆內完成 draft→critique→v2 |
| 外部依賴 | 無 | `scripts/cl-flow-runner.js` + `GEMINI_API_KEY` + artifacts 資料夾 | 無 |
| 自我批評 | ❌（`rp.md` 明文：精煉階段無參照物，強制批評是表演） | ✅（Verdict 層） | ✅（本指令核心步驟） |
| 8維度掃描 | ✅ 完整（`<architecture_scan>`，本指令沿用同一份定義，不重複維護語義） | 間接透過 /rp Step 0 | ✅（Step 4，直接引用 `rp.md` 的 8 個維度標籤） |
| 適合場景 | 任務模糊，先整理思路 | 需要具體實作步驟+執行授權 | **大型架構/治理決策**，想要快速拿到自我審過的 v2 版本，不需要外部工具或正式 Gate |

**8 維度的唯一定義來源是 `.fhs/ai/commands/rp.md` 的 `<architecture_scan>` 區塊**——本檔案 Step 4 只引用維度標籤名稱，不重複撰寫語義說明，避免兩處各寫一份定義而漂移（同 `canonical_keys.yml` 精神）。若未來要新增第 9、10 個維度，改 `rp.md` 那一處即可，`/8d` 本身不需要跟著改名（見下方「命名說明」）。

## 執行步驟

### Step 1 — 識別對象

若 `/8d` 後有參數，該參數即為分析對象；若無參數，取對話中**最近一則** AI 提案/方案（通常是回覆結尾的建議段）作為對象。若找不到明確對象，回報「請指明要分析的方案」並停止。

### Step 2 — 實施草案 → 實施計畫草案

先給出對象的具體實施草案（做法本身），再展開成有步驟的實施計畫草案（做法+順序+涉及檔案）。若對象已經是計畫形式（例如上一則回覆已列步驟），可直接沿用，不必重寫一份幾乎相同的文字。

### Step 3 — 自我批評（3個弱點）

對 Step 2 的計畫草案，找出**恰好 3 個**具體弱點（不多不少；若真心找不到 3 個實質問題，明言「僅 N 個」並說明為何，不湊數）。每個弱點須指出：哪裡壞、什麼情況下會壞。

### Step 4 — 更好的版本 v2

修正 Step 3 三個弱點後的版本。v2 完成後，逐一套用 `rp.md` `<architecture_scan>` 的 8 個維度標籤，對 v2 判定 `[相關]` 或 `[N/A]`（沿用 rp.md 既有的「強制/N-A」慣例，不新造判定語彙）：

```
perf / ux_mgmt / conflict / token / long_term / responsive / subagent_skill / history
```

### Step 5 — 三平台顧慮（本指令獨有，rp.md 8維度不含此項）

v2 是否涉及 Claude Code 治理層/hook/skill/subagent 設計時，額外檢查三個外部工具的相容性顧慮，逐一列 `[相關]`/`[N/A]`：

- **Antigravity**：是否假設了 Claude Code 專屬機制（PreToolUse hook、`.claude/` 目錄慣例）而在 Antigravity/Gemini 環境不成立
- **VSCode**：透過 VSCode 內嵌 Claude Code 使用時是否有路徑/終端差異
- **Codex**：若方案涉及可攜出去給其他 agent 用的知識（非 FHS 專屬邏輯），是否已寫成中立格式而非綁死 Claude 專屬語法

若 v2 純屬業務邏輯（例如訂單欄位、成本計算），三平台顧慮全部 `[N/A]`，一行帶過即可，不必展開論述。

### Step 6 — 輸出

單則回覆內依序輸出 Step 2→5 全部內容，最後一行明確給出 v2 是否已可交付執行的裁決（可交付 / 需 Fat Mo 再確認 X 點）。**不停等額外 Gate**——若 v2 已可直接動手，接下來的實際改動仍照 CLAUDE.md Rule 3 走（架構改動先提方案等確認；小改動可直接做）。

## 命名說明（回應「甚至叫9-10d」的顧慮）

「8d」是**沿用 rp.md 現行 8 維度數量的口語代稱**，不是鎖死的版本號。維度定義本身住在 `rp.md`，`/8d` 只是引用；若日後 `rp.md` 的 `<architecture_scan>` 擴充到 9 或 10 個維度，`/8d` 這個指令名稱**不需要跟著改**（改名成本低但無實益——名稱是肌肉記憶，不是精確計數器）。真要重新命名，直接 `git mv` 這兩個檔案即可，無其他檔案引用此指令名稱，零連鎖修改。
