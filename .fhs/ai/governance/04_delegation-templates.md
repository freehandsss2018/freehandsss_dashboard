# 04 — 派工 Prompt 模板（填空即用）

> **Version**: v1.0.1（2026-07-04，Session 137；同日對抗審查修正：T1 巨檔警語明確化、T5 回報上限放寬至 60 行）
> **用法**：選對應任務型態的模板 → 填 `{...}` 空格 → 整段作為 Agent tool 的 `prompt`。`model` 與 `subagent_type` 按模板頭建議帶入（分派邏輯見 `02 §4`）。
> **鐵律**：三件套（目標動機/驗收條件/回報格式）不可刪節。subagent 是冷啟動，「如前所述」「你知道的那個檔案」= 必然失敗。

---

## T1 — 搜尋 / 定位（subagent_type: Explore；model: sonnet，簡單題 haiku）

```
【目標】在本 repo 找到 {要找的東西：函式定義/字串來源/所有引用點}。
【動機】主任務是 {一句話}，需要先確定 {位置/影響面} 才能安全動手。
【範圍】搜索廣度：{quick=單點查找 / medium=中度探索 / very thorough=跨多處命名慣例}。
起點線索：{已知的關鍵詞/檔名/相關路徑}。
排除：{不用看的目錄，如 node_modules、.fhs/memory/archive}。
⚠️ 巨檔紀律：Freehandsss_Dashboard/ 目錄下所有 .html（含 current/V41/V42）、handoff.md、CHANGELOG.md 禁全讀，只准 Grep+窗口讀（≤250行）。
【驗收】每個結果給出 檔案路徑:行號 + 該行原文引用；若找不到，列出已嘗試的 ≥3 組關鍵詞與已排除的位置。
【回報格式】≤30 行：結果清單（路徑:行號+一行說明）→ 有無意外發現（≤3條）→ 信心：高/中/低+一句理由。禁貼大段原文。
```

## T2 — 實作（subagent_type: general-purpose；model: sonnet）

```
【目標】實作 {功能一句話}。
【動機】{為什麼要做；操作者/系統得到什麼}。邊界情況取捨原則：{如「寧可少做不可猜」}。
【上下文】相關檔案：{路徑清單+各自角色}。既有慣例：{要模仿的現有實現位置 檔案:行號}。
【硬約束】{本任務適用的紅線，至少檢查：禁改 HTML ID / 禁動 captureFormState / 財務欄位職責分工 / n8n Code Node 回傳 [{json:{...}}]}。
【驗收條件】（逐條可機械判定）
1. {行為斷言，如：呼叫 X(a,b) 回傳 c}
2. {運行證據要求，如：node --test 通過 / mock 請求回 201}
3. 巨檔改動：替換三步計數證據（改前 grep -c =1、改後 舊0新1）。
【回報格式】≤40 行：改了哪些檔（路徑:行號範圍+各一行說明）→ 驗收條件逐條附證據（引用命令輸出）→ 未處理的邊界情況 → 信心：高/中/低+理由。diff >50 行落檔傳路徑。
```

## T3 — 重構（subagent_type: general-purpose；model: sonnet；模式驗證後批次可降 haiku）

```
【目標】重構 {對象}：{從什麼形態 → 到什麼形態}。行為必須完全不變。
【動機】{為什麼值得動：重複/易錯/阻礙什麼}。
【範圍】只動：{明確檔案/函式清單}。範圍外即使看到問題也不改，記入回報「發現但未動」。
【安全網】動手前先確認行為基準：{既有測試命令 / 若無測試，先為受影響路徑寫最小 smoke test 或記錄現值}。
【驗收條件】
1. 基準驗證改前改後輸出一致（附兩次輸出）。
2. 引用點全數更新：grep 舊符號 =0（附計數）。
3. 無 scope creep：diff 只含【範圍】內檔案。
【回報格式】≤40 行：改動摘要 → 基準對照證據 → grep 計數 → 「發現但未動」清單 → 信心+理由。
【批次降級用法】若重構含 N 處同型替換：先手工做 2 處並驗證 → 把「精確替換規格」（舊字串/新字串/預期匹配數）寫成清單 → 另派 haiku 批次執行 → 派工方抽樣 ≥10% 驗證。
```

## T4 — 研究（subagent_type: general-purpose；model: sonnet；含網頁檢索）

```
【目標】回答 {研究問題，一句話}。
【動機】{答案將決定什麼決策}。所以重點是 {決策需要的維度：成本/相容性/風險…}。
【方法要求】優先序：官方文檔 > 官方 repo/changelog > 高質量社群來源。每個關鍵結論給來源 URL。區分「文檔說的」vs「你推斷的」，推斷要標明。
【深度預算】最多 {N} 個來源 / {M} 輪檢索；到預算即停，回報已知+未知，不無限深挖。
【驗收條件】研究問題有直接回答（或明確說「查不到，因為…」）；關鍵版本號/API 名稱給出處；與 FHS 環境的適用性有一段評估（{本環境相關約束，如 NAS n8n Code Node 無 fetch}）。
【回報格式】≤35 行：直接答案（1-3 句）→ 依據（來源+要點）→ 對本環境的適用性/風險 → 未能確認的部分 → 信心+理由。完整筆記落檔 {scratchpad 路徑} 傳路徑。
```

## T5 — 審查（subagent_type: general-purpose 或 code-reviewer；model: 常規 sonnet / 財務·schema·部署類 opus）

```
【目標】對抗式審查 {對象：diff/檔案/方案}，找出真問題。你不是來蓋章的：找不到任何問題時，必須列出你查過哪些角度（≥5 個）證明找過。
【動機】此改動將 {影響面：上生產/寫財務欄位/改 schema}，錯誤成本 {高/中}。
【審查基準】對照：{AGENTS.md 硬規則清單相關節 / 驗收條件原文 / 02 §5 分流表}。
【必查角度】1.與既有規則衝突 2.路徑/工具名/ID 是否真實存在（逐一驗證，不信文中自稱）3.邊界與失敗路徑 4.弱模型會誤讀的模糊語句（審制度檔時）5.{本次特有角度}。
【驗收條件】每個 finding 附：位置（檔案:行號）+ 問題一句話 + 失敗場景（什麼輸入/狀態下爆）+ 建議修法。無失敗場景的 finding 標「風格建議」分開列。
【回報格式】≤60 行（審查類回報允許比其他模板長，findings+已查角度佔行數）：verdict（PASS / PASS-with-fixes / FAIL）→ findings 按嚴重度排序 → 風格建議（≤5條）→ 已查角度清單 → 信心+理由。
```

---

## 通用尾註（每個模板派出前自查 10 秒）

- [ ] 路徑全部是絕對或 repo 相對真實路徑（不是「那個檔案」）
- [ ] 驗收條件每條都能機械判 PASS/FAIL
- [ ] 回報有行數上限 + 長產物落檔指示
- [ ] 巨檔紀律警語在（凡任務可能碰 HTML/handoff/CHANGELOG）
- [ ] model 按 02 §4 分派表選定（有懷疑就 sonnet）
