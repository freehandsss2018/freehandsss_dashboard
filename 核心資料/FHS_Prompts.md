# 📚 FHS 全域開發指示詞庫 (FHS Prompts Library)
> **[給 Antigravity 的系統指令]**：當你在 `.cursorrules` 中觸發了「動態情境路由協議」時，請在此文件中尋找對應的【情境】，並嚴格套用其下的「執行授權與約束」。

---

## 👑 【情境一：任務完結與全局文檔同步】
**觸發時機**：當一個功能開發完成、Bug 修復完畢，準備收尾時。
**執行授權與約束**：
1. **更新 Changelog**：請在 `Changelog.md` 中新增一筆紀錄，總結本次改動，並自動推進微小版本號 (例如 V31.x)。
2. **藍圖核對**：請掃描剛才的代碼改動，判斷是否影響了 `FHS_Blueprint.md` 中的「資料結構」、「1:1 映射定律」或「財務邏輯」。若有影響，請同步更新藍圖內容。
3. **註解檢查**：確保新增的 HTML/JS 函數或 n8n 節點都有清晰的繁體中文註解。
4. **狀態回報**：完成上述同步後，請向我輸出一段簡短的「Fat Mo 專屬戰報」，列出已更新的文件與版本號。

---

## 🎨 【情境二：前端 UI 修改與自動 QA 測試】
**觸發時機**：當要求修改 Dashboard 介面、新增表單欄位、或調整版面樣式時。
**執行授權與約束**：
1. **視覺約束**：嚴守 V31.0 玻璃擬態 (Glassmorphism) 設計。全域核對中心的表格強制使用 `<td rowspan>` 與 `vertical-align: top;`，網格內字體 13px、Header 12px 加粗。
2. **測試閉環**：代碼修改完成後，不要問我，請直接切換至「沙盒模式」，並透過腳本或模擬執行 `runAllAudits()`。
3. **修復授權**：若 QA 日誌返回 ❌，授權你自行分析錯誤並連續進行最多 5 次的自我修復嘗試。只有輸出『✅ [PASS]』時，才停止迴圈並回報。

---

## ⚙️ 【情境三：後端 n8n 邏輯修改與映射對齊】
**觸發時機**：當要求修改 n8n 節點邏輯（如 `FHS_Core_OrderProcessor`）或調整資料流時。
**執行授權與約束**：
1. **防爆約束**：嚴格遵守「非阻塞原則」，輔助邏輯出錯不可中斷主線訂單。
2. **1:1 映射定律**：檢查前端送來的 payload 與 Airtable Schema 是否完全對齊。
3. **防重複機制**：任何寫入 Airtable 的動作，必須具備 Upsert 邏輯。
4. **驗證要求**：修改完成後，請提供一段測試用的 JSON Payload 給我，讓我可以在前端沙盒中發射測試。

---

## 👩‍💼 【情境四：Ling Au 專屬防呆 UI 與體驗優化】
**觸發時機**：針對前線人員 (Ling Au) 的操作介面進行防呆強化或流程優化時。
**執行授權與約束**：
1. **零容錯設計**：絕不能依賴使用者的自律。必須實作嚴格的格式驗證 (Validation)、必填鎖定，或將 Input 改為下拉選單/Radio Button。
2. **白話文提示**：所有的報錯提示框 (Alert / Toast) 必須使用溫和、口語化的繁體中文，絕對禁止出現工程報錯代碼。
3. **視覺防護**：當欄位填寫錯誤時，需加入明顯的紅色邊框或微小的抖動動畫 (Shake animation) 提醒。
4. **極端測試**：修改後，開啟沙盒模式模擬「不小心亂按」的極端操作進行壓力測試。

---

## 🚨 【情境五：極速重構與全自動除錯迴圈】
**觸發時機**：當系統發生未知錯誤、終端機噴錯，或 Fat Mo 要求「直接看報錯修到好」時。
**執行授權與約束**：
1. **全權委託**：授權你直接修改相關代碼 (HTML/JS/n8n JSON)，無需事先請示每一行改動。
2. **靜默執行**：執行測試腳本強制加 `-y` 參數。生成臨時除錯腳本務必包含結尾自刪邏輯 (`os.remove(__file__)`)。
3. **自我修復迴圈**：修改後立即觸發測試。若遇錯誤，自行閱讀 Error Log 並進行最多 5 次連續迭代修復。
4. **終止條件**：直到全數 ✅ [PASS] 才停止並總結解法。若 5 次仍失敗，暫停並列出卡關原因。

---

## 🗄️ 【情境六：資料庫擴充與全端映射對齊】
**觸發時機**：當需要在 Airtable 增加新欄位（例如新增費用、產品線），牽一髮動全身時。
**執行授權與約束**：
1. **UI 擴充**：在 Dashboard 加入對應欄位，符合視覺與防呆規範。
2. **Payload 擴充**：確保欄位正確封裝進 `sendOrder` JSON Payload。
3. **n8n 節點對齊**：提議修改對應的工作流（如 `FHS_Core_OrderProcessor`），確保 Webhook 能解析並 mapping 到 Airtable。
4. **藍圖註冊**：將新欄位邏輯補充進 `FHS_Blueprint.md`。請先列出「受影響檔案清單與計畫」讓我確認。

---

## 🛡️ 【情境七：API 防護與效能邊界測試】
**觸發時機**：針對系統進行壓力測試，或驗證防抖 (Debounce) / 佇列 (Queue) 機制時。
**執行授權與約束**：
1. **模擬高壓**：撰寫臨時腳本模擬短時間內大量 API 請求。
2. **防護驗證**：檢查系統是否成功攔截多餘請求 (如 800ms 防抖)，Airtable 是否只收到合理的批次請求。
3. **雲端之眼測試**：確保超出負載的錯誤能被 `FHS_System_ErrorMonitor` 捕捉並寫入 Error_Logs，前端不崩潰。
4. **清理回報**：測試完畢後回報阻擋率，並刪除臨時腳本。
